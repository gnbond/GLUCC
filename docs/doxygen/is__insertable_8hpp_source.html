16/*
17 * Older LLVM libcpp ostreams versions (in particular, as shipped with the clang
18 * version in MacOs XCode 10) have a bug that means the usual C++17 SFINAE idiom
19 * does not work correctly for std::ostream.
20 *
21 * On these systems, `decltype(std::declval<std::ostream>() <<
22 * std::declval<T>())` does not work as expected if `T` is not an insertable
23 * type. This (correctly) fails outside of declval(), but the template
24 * `operator<<(Stream&&, const T&)` this is ultimately referencing is not
25 * sufficiently constrained so inside decltype() the SFINAE does not correctly
26 * propagate up to the template specialization. End result is that the
27 * resulting trait gives false positives for std::ostream.
28 *
29 * So far I have found this bug in libcpp shipped with XCode 10 (v 6000) and
30 * also in a homebrew-installed llvm11 (v11000), but it is not present in XCode
31 * 14. We detect that here and use an alternate idiom for those compilers.
32 *
33 * As far as I can see, gcc's libstdc++ does not suffer from this problem.
34 *
35 * See
36 * https://github.com/llvm/llvm-project/commit/fdc41e11f9687a50c97e2a59663bf2d541ff5489
37 */
38

40#define BROKEN_LIBCPP
